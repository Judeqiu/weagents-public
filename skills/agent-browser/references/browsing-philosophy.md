# 浏览哲学 - 像人一样思考

> 移植自 web-access skill 的核心哲学

## 核心原则

**像人一样思考，兼顾高效与适应性的完成任务。**

执行任务时不会过度依赖固有印象所规划的步骤，而是带着目标进入，边看边判断，遇到阻碍就解决，发现内容不够就深入——全程围绕「我要达成什么」做决策。

---

## 四步框架

### ① 拿到请求 — 明确成功标准

先明确用户要做什么，定义成功标准：
- 什么算完成了？
- 需要获取什么信息？
- 需要执行什么操作？
- 达到什么结果？

这是后续所有判断的锚点。

### ② 选择起点 — 验证而非规划

根据任务性质、平台特征、达成条件，选一个最可能直达的方式作为第一步去验证。

**不要预设计划，而是快速验证：**
- 一次成功当然最好
- 不成功则在③中调整

**选择逻辑：**

| 场景 | 推荐起点 |
|------|----------|
| 需要操作页面、需要登录态 | agent-browser CDP |
| 已知静态方式不可达的平台 | agent-browser CDP (跳过静态层) |
| 文章、博客、文档 | Jina 预处理 (省 token) |
| 简单信息查询 | curl 或 WebSearch |

### ③ 过程校验 — 边做边看

每一步的结果都是证据，不只是成功或失败的二元信号。

**用结果对照①的成功标准，更新判断：**
- 路径在推进吗？
- 结果的整体面貌（质量、相关度、量级）是否指向目标可达？

**调整策略：**
- 发现方向错了立即调整
- 不在同一个方式上反复重试
- 搜索没命中不等于"还没找对方法"，也可能是"目标不存在"
- API 报错、页面缺少预期元素、重试无改善，都是在告诉你该重新评估方向

**障碍处理：**
- 遇到弹窗、登录墙等障碍
- 判断它是否真的挡住了目标
- 挡住了就处理，没挡住就绕过
- 内容可能已在页面 DOM 中，交互只是展示手段

### ④ 完成判断 — 确认而非完美

对照定义的任务成功标准，确认任务完成后才停止。

**不要过度操作**，不为了"完整"而浪费代价。

## 工具选择哲学

### 三层通道调度

```
┌─────────────────────────────────────────────────────────────┐
│  第一层: 信息发现                                            │
│  - WebSearch: 发现信息来源、获取关键词结果                    │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  第二层: 轻量获取                                            │
│  - curl: 原始 HTML，快速简单                                 │
│  - Jina: 文章转 Markdown，大幅省 token                        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  第三层: 浏览器自动化                                        │
│  - agent-browser: 动态页面、反爬平台、需要交互                │
└─────────────────────────────────────────────────────────────┘
```

### 具体选择指南

| 场景 | 工具 | 原因 |
|------|------|------|
| URL 未知，需要搜索 | WebSearch | 发现信息入口 |
| URL 已知，文章/文档 | Jina | 省 token，结构化输出 |
| URL 已知，原始 HTML | curl | 快速，获取 meta/JSON-LD |
| 反爬平台 (小红书/微信/知乎) | agent-browser | 动态渲染，模拟真实用户 |
| 需要登录态 | agent-browser | 复用 session |
| 需要交互操作 | agent-browser | 点击、填写、滚动 |

### Jina 积极推荐

**明确鼓励在合适场景主动使用 Jina 节省 token。**

**适合 Jina：**
- 文章、博客、新闻
- 文档、README
- 教程、指南

**不适合 Jina：**
- 数据面板、商品页
- 搜索结果页
- 需要交互的动态页面

## 浏览器操作哲学

### 在页面内自由导航

进入浏览器层后，就是你探索的空间：

- **看**: 用 `snapshot` 或 `eval` 查询 DOM，发现页面上的链接、按钮、表单、文本内容
- **做**: 用 `click`、`fill`、`scroll` 在页面内自然导航
- **读**: 用 `get text` 或 `eval` 提取文字内容

**先 eval 了解页面结构，再决定下一步动作**

看到列表就点进详情，看到分页就翻页，看到内容就提取。不需要提前规划所有步骤。

### 程序化 vs GUI 交互

| 方式 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 程序化 | 速度快、精确 | 易被识别为爬虫 | 简单页面、已知结构 |
| GUI 交互 | 确定性高、像真人 | 步骤多、速度慢 | 反爬平台、复杂交互 |

**根据对目标平台的了解来判断。** 当程序化方式受阻时，GUI 交互是可靠的兜底。

**注意:** 不要凭记忆构造带参数的 URL——内部参数（城市代码、分类 ID 等）随时可变，不确定时通过页面交互或从 DOM 中提取实际链接。

## 并行分治策略

任务包含多个**独立**调研目标时，合理分治给子任务并行执行。

### 好处

- **速度**: 多任务并行，总耗时约等于单个子任务时长
- **上下文保护**: 抓取内容不进入主上下文，主任务只接收摘要，节省 token

### 分治判断标准

| 适合分治 | 不适合分治 |
|----------|-----------|
| 目标相互独立，结果互不依赖 | 目标有依赖关系，下一个需要上一个的结果 |
| 每个子任务量足够大 | 简单单页查询，分治开销大于收益 |
| 需要浏览器或长时间运行的任务 | 几次 WebSearch / Jina 就能完成的轻量查询 |

### 使用方式

```bash
# 并行调研多个产品
python3 parallel_research.py --targets '[
    {"name": "产品A", "url": "https://a.com", "task": "提取价格"},
    {"name": "产品B", "url": "https://b.com", "task": "提取规格"}
]'
```

## 站点经验积累

### 经验的价值

操作中积累的特定网站经验，按域名存储，跨 session 复用。

**已有经验的站点：** 查询 `site_experience.py list`

### 使用经验

确定目标网站后，如果有匹配的经验，必须读取对应文件获取先验知识：
- 平台特征
- 有效模式
- 已知陷阱

### 更新经验

CDP 操作成功完成后，如果发现了有必要记录的新站点或新模式，主动写入对应的站点经验文件。

**只写经过验证的事实，不写未确认的猜测。**

```bash
# 查询站点经验
python3 site_experience.py query xiaohongshu.com

# 添加新的站点经验
python3 site_experience.py init new-site.com --alias "别名"
python3 site_experience.py add new-site.com --feature "需要登录"
```

## 登录判断

**目标内容拿到了吗？** 这是登录判断的核心问题。

打开页面后先尝试获取目标内容。只有当确认**目标内容无法获取**且判断登录能解决时，才告知用户：

> "当前页面在未登录状态下无法获取[具体内容]，请在你的 Chrome 中登录 [网站名]，完成后告诉我继续。"

**登录完成后无需重启任何东西**，直接刷新页面继续。

## 总结: Skill = 哲学 + 技术事实

不是操作手册，而是讲清 tradeoff 让 AI 自己选，不替它推理。

**核心记住：**
1. 目标导向而非步骤驱动
2. 边看边判断，遇到阻碍就解决
3. 过程校验，及时调整方向
4. 确认完成标准，不过度操作
